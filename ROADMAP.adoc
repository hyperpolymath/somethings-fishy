= Something's [Robo] Fishy — Roadmap
:toc: left
:icons: font

Inspired by perplexity-based "AI text detectors" being broken by design, this
project aims at a valid and reliable hybrid of Turnitin-style authorship
attribution and autopsy-style forensic reconstruction — useful to both
educators and developers — with a long tail of capabilities extending into
multi-agent interaction reconstruction and pseudointent inference.

See `docs/decisions/0001-purpose-and-scope.adoc` for the full design rationale.

== Capability ladder

Each stage is additive: later stages depend on the reference corpus and
labelled observations generated by earlier stages.

=== v0 — Observe (in progress)

Shallow-signal scanners in clone-first isolation, feature-emitting A2ML
reports. No classifier. No VeriSimDB integration yet.

Goals:

* Clone a target into `/mnt/eclipse/robofishy-scenes/<ts>-<slug>/`.
* Run detectors for: agent-addressed files, bot commit trailers, banned-
  pattern sprays, `panic-attack` subprocess invocation.
* Emit S-expression A2ML report to the scene directory.
* Every finding carries a stable content-addressed ID so repeated runs are
  test-retest-reliable (feeds the later validity pipeline).
* Hard invariants enforced: never write to the subject, scanners only read.

Non-goals at this stage:

* No classifier, no confidence scores beyond "present/absent".
* No VeriSimDB writes yet (scene directory is the evidence store).
* No multi-target parallelism (chapeliser integration deferred).
* No feedback-o-tron integration yet (nothing to emit changes through).

=== v1 — Detect

Calibrated multi-class attribution with open-set rejection. Educators need
this to be honest about uncertainty; developers need it to be fast.

Goals:

* Structural + temporal feature extraction modules riding on top of v0
  scanners. Each scanner emits `(finding, feature_vector)` tuples.
* *RBF-kernel classifier* (`linfa` + `linfa-svm`) trained on user-declared
  crime-scene labels. Pure Rust, runs on a laptop, explainable via support
  vectors.
* *Open-set rejection*: if nearest-match confidence falls below a
  threshold, emit "unknown" rather than forcing a label.
* *Conformal prediction* wrapper to produce calibrated confidence
  intervals, not point estimates. This is what "valid and reliable"
  actually demands.
* Crime-scene declaration syntax (A2ML): user marks commits, ranges, or
  file regions as ground truth for specific agents.
* VeriSimDB wiring: two schemas (`agent_reference_corpus`,
  `investigation_evidence`) in a single robofishy instance.

=== v2 — Reverse

Propose reversals for damaged ranges. No autonomous writes — every proposal
goes through feedback-o-tron for human adjudication.

Goals:

* Damage classification: identify the edit windows that look like agent
  damage (drawn from v1 classifier output + crime-scene patterns).
* Reversal proposal generator: produces a candidate reverse-patch per
  damage window.
* Safety guard: reversal proposals must preserve unrelated intervening
  changes; the reversal is a surgical revert, not a hard reset.
* Feedback-o-tron integration: all proposals emitted as adjudication tasks.

=== v3 — Repair

Beyond reversal: actively propose repairs for recognised damage types.

Goals:

* Damage-type library (growing as observations accumulate): "half-deleted
  function", "stub inserted and forgotten", "test disabled and not
  re-enabled", "type assertion sprayed to silence the checker", …
* Repair template library, one per damage type.
* Suggestions still routed exclusively through feedback-o-tron.

=== v4 — Interact (multi-agent reconstruction)

Detect and reconstruct multi-agent sessions: when Claude handed off to
Gemini mid-session, when Codex took over from Vibe, what each agent
contributed.

Goals:

* Change-point detection on per-commit attribution time series (v1
  classifier produces the signal; v4 segments it).
* Handoff signature library: stylistic discontinuities, context-window
  resets, comment-addressed handovers ("continuing from where Claude left
  off").
* Interaction graph output: directed graph of agent → agent transitions
  with timestamps and scope.

=== v5 — Pseudointent (inferring what they were trying to do)

The hardest stage. Given an observed action sequence, infer the latent goal
the agent was pursuing — and detect goal drift when the agent went AWOL
(started chasing X, ended up doing Y).

Research directions (pick one or combine):

* *Symbolic abduction* over a hand-curated goal library (refactor, fix
  compile error, disable failing test, satisfy linter, paper over a merge
  conflict, …). Most explainable; brittle until the library is broad.
* *Inverse reinforcement learning*: learn a reward function that would
  make the observed behaviour optimal. More flexible, data-hungry, opaque.
* *LLM-as-hypothesizer with symbolic scoring*: generate candidate goals
  with a model, score each against observed actions with non-LLM metrics.
  Avoids the trust-loop if scoring is fully auditable.

Early instinct: start with symbolic abduction over a small goal library
and grow from there. Escalate to IRL only when the library stops covering
observations.

=== Ongoing — Validate

Psychometric validity work, not a one-time milestone. Required because the
Turnitin lineage means the tool must defend false-positive rates to
real-world auditors.

Goals:

* Labelled ground-truth set, grown from crime-scene declarations.
* Construct validity studies: does "this is Claude" measure what we claim
  it measures?
* Content validity: do our features actually cover the construct?
* Criterion validity: do flagged damage reports predict real downstream
  problems?
* Test-retest reliability: same input → same output (enforced by stable
  finding IDs from v0 onwards).
* Inter-rater reliability: cross-validation folds, held-out agents.
* Calibrated false-positive budgets, documented per audience (educators
  get tighter budgets than developers).

== Audience fronts

One detection engine, two audience modes. These should become subcommands
or profiles over the same core — not separate binaries.

=== Educators

* Primary use: coursework authenticity assessment.
* Critical requirement: *per-student baseline* — "is this anomalous *for
  this student*?" is a one-class detection problem, separate from the
  multi-class "which bot is this?". Both live in this tool.
* False-positive cost is very high (accusing an innocent student).
  Confidence thresholds must be conservative; the tool should prefer
  "insufficient evidence" to any assertion.
* Reports must be defensible to an academic appeals process.

=== Developers

* Primary use: post-hoc forensics on a repo or estate.
* Chapeliser-distributed batch runs over many repos.
* Higher tolerance for false positives (cost is a wasted investigation,
  not a wrongful accusation).
* Prioritised findings, triage-oriented UI.

== Explicit non-goals

These are *not* what robofishy is for:

* Not a generic repo linter or RSR-compliance checker.
* Not an auto-remediation tool (writes only ever via feedback-o-tron).
* Not a perplexity-based "AI text detector" — that approach is structurally
  broken and inspired this project by counter-example.
* Not a human-vs-AI binary detector. The target space is specific agents
  (Claude, Gemini, Codex, Vibe, …) with open-set rejection for "unknown"
  and "likely human".
* Not a trust-loop using an LLM to judge LLM output. Any LLM-as-hypothesizer
  use must be paired with non-LLM scoring.
